## 2026-07-01 — Routing/synthesis agent built

**Accomplished this session:** Built the routing/synthesis layer (architecture doc Section 5-6), the biggest remaining piece of net-new logic flagged in the prior handoff. Confirmed approach with user before writing code via 3 explicit design decisions (see below), then implemented and validated end-to-end against real D1/Vectorize/Claude.

**Design decisions (all user-confirmed, recommended options taken):**
1. **Routing**: reuse each specialist's own model/product detector against every sheet — no extra Claude call. A sheet can match 0, 1, or multiple brands. 0-match sheets are flagged "unclassified", not dropped.
2. **Specialist call granularity**: batched per brand, not per sheet. Sheets are grouped by matched brand and each relevant specialist is called once with all of that brand's matched sheets concatenated (tagged by sheet number), so a specialist reasons across all its sheets together and API calls scale with brand count, not sheet count.
3. **Synthesis**: pure deterministic code, no Claude call. Findings already carry `finding_type` (one of the six sections); synthesis just buckets, dedupes, and orders by brand — zero hallucination risk since nothing is paraphrased away from its citation.
4. **Cross-brand sheets**: sheets matched to 2+ brands get their findings re-grouped into a deterministic "Cross-Brand Watch" callout (no extra Claude call) — this is the multi-product cross-checking gap the architecture doc explicitly flagged as unsolved (line 130), now addressed at the re-grouping level rather than a new reasoning pass.

## 2026-07-01 (cont'd) — Print-reading pipeline: extraction, tag correlation, and reliability guards

**Why this session's focus shifted here:** user flagged that the routing/synthesis agent is only as good as its input — real commercial print sets need to actually be *read* (structured extraction + cross-sheet correlation) before any of Section 5-6 matters. User's framing: "if we do not get this step right, nothing really matters. The output will be incomplete and not usable." Treated as the highest-priority remaining gap over persistence/frontend.

**Design established (user-confirmed):**
- The hard problem isn't OCR quality, it's **cross-sheet correlation**: a brand's supporting sheets (structural steel, electrical panel, RCP obstruction) routinely never mention the brand/product by name, so naive per-sheet keyword routing misses them. Real PMs correlate via shared gridlines/tags/cross-references instead.
- Two-stage pipeline: (1) per-sheet extraction via Claude reading native PDF pages directly (not OCR, not pre-extracted text) into structured JSON (schedules, dimensions, keynotes, tags, gridlines, cross-references, full rawText); (2) deterministic (no extra Claude call) tag/gridline index used to correlate sheets one-hop only — a sheet is "inferred" relevant to a brand if it shares a tag/gridline with a directly-matched sheet, never chained further, and always labeled inferred vs. direct so it stays auditable.
- Test file: real project PDF, Bryant & Stratton Parma OH renovation. Two candidate files existed locally — the 273-page **Project Manual** (CSI spec text, not what we needed) vs. the 62-page **COMPLETE Bid Set** (actual 24"x36" drawing sheets, 44MB) in `~/Downloads`. Used the Bid Set.
- Discovered Claude's PDF input caps at 32MB/100 pages — the 44MB/62-page Bid Set already exceeds the byte cap even though page count is fine, confirming per-sheet splitting (via `pdf-lib`, pure JS, no system deps) is mechanically required, not just an architectural preference.

**Implementation:**
- `src/lib/pdfSplit.js` — `splitPdfIntoPages` (pdf-lib, one PDF per page) + `batchPages` (groups pages for extraction calls respecting both a page-count cap and a byte-size cap).
- `src/lib/extraction.js` — `extractSheetsFromPdf`. Sends each batch as multiple `document` content blocks in one Claude call, discipline-agnostic extraction prompt (no brand-bounding, unlike specialists). Hit a real bug on first real-data run: `max_tokens: 8192` was too small for text-dense sheets (a general-notes page alone blew the budget) and the response got truncated mid-JSON. The generic "retry and ask for valid JSON" logic (borrowed from the specialists' inch-mark-quoting fix) doesn't work for truncation — it just truncates again. Fixed properly: detect `stop_reason === 'max_tokens'` specifically, and on truncation *split the batch in half and recurse* (not blind retry); for a single page that still truncates alone, escalate max_tokens (16k → 32k → up to 64k) before giving up. Verified working on real data: a 6-page batch truncated, correctly split to 3+3, then 3 split to 2+1, then the single dense page (A-002, accessibility requirements) escalated to 32k tokens and succeeded.
- `src/lib/tagIndex.js` — `buildTagIndex` (deterministic map of tag/gridline/cross-reference → sheet numbers) and `correlateOneHop` (direct-match sheets + one-hop inferred sheets, with `inferredVia` tracking *why* each inferred sheet was pulled in, for auditability).
- `src/lib/extractionGuards.js` — added after real-data testing exposed two specific, recurring failure modes (not random hallucination, identifiable patterns):
  1. **Repeated graphical symbols read as literal tags.** On sheet A-004 (a real floor plan), the token "6578" appeared 68 times in the raw transcription and got extracted as if each occurrence were a distinct tag — almost certainly a repeated hatch/fixture symbol, not 68 real identifiers. `filterNoisyTags` drops any tag occurring more than a threshold (default 6) times on its own sheet, and logs why.
  2. **Multi-tier schedule tables lose column/row alignment.** A "Plumbing Fixture Analysis" schedule declared 11 columns but several rows returned 13 values (source table almost certainly has a merged/grouped header that didn't flatten correctly). `validateSchedules` doesn't try to auto-correct (too risky — guessing which column absorbed the extra value could silently mislabel real numbers) — it flags the schedule with `columnMismatch: true` and a warning so nothing downstream cites unverified values from it.
  - Both guards verified against the real saved extraction output before re-spending API calls: correctly dropped the 68x-repeated tag while keeping all 39 legitimate room tags (`131 LACTATION`, `128 MECH.`, etc.), and correctly flagged both mismatched schedules.
- `src/lib/test-extraction.js` + `npm run test:extraction <pdf> [maxPages]` — runnable end-to-end test, prints per-sheet summary, extraction warnings, tag index shared-key report, and a sample one-hop correlation.

**Reliability assessment (user asked directly whether AI print-reading is reliable — answered from actual evidence, not speculation):** straightforward transcription (general notes, legends, room tags, prose calculations) came through excellently on real data. Complex multi-tier tables and repeated graphical symbols are where it currently breaks, but in identifiable, guardable ways — not randomly. Also: this system's hardest-to-verify content (exact tolerances/formulas) was never going to come from runtime print-reading anyway — that ground truth already lives in the hand-verified D1 parameter store. What the print needs to reliably supply is narrower (what was specified, where, cross-references) and squarely in the "already works well" bucket.

**Full 62-page run results (real data, not a sample):**
- 47 extraction warnings fired correctly across 62 sheets — genuinely messy real-world tables (multi-tier door schedules, electrical panel schedules, HVAC sizing summaries) correctly flagged `columnMismatch` rather than silently trusted; repeated symbols correctly dropped as noisy tags on the sheets where they were dense (e.g. code-related numbers like "703", "610" repeating dozens of times on the accessibility-requirements sheet).
- Every sheet's title/discipline/schedule/rawText content came through cleanly — no wholesale extraction failures across the real set.

**Design points:**
- **"Never invents" is enforced deterministically, not just prompted.** The reconciler assigns stable ids (F1…Fn) to the deduped findings, and `applyReconciliation()` validates every id the model returns against the real finding list — unknown ids, self-merges, double-removals, and escalations of non-ABSENCE findings are dropped with guard warnings, never trusted. The model proposes; the guard disposes.
- **No paraphrasing, ever** (same philosophy as findingsGuard/extractionGuards): a corroboration merge keeps one finding's text verbatim and attaches the merged findings in full as `corroborated_by` (+ optional `corroboration_note`); culled findings are recorded in full on `reconciliation.culled`, not deleted; derived-value flags annotate (`derived_value_flag`) without removing. The acceptance test asserts every surviving description matches an input verbatim.
- **Failure degrades, doesn't destroy:** if the reconciler call throws, the report ships un-reconciled with `reconciliation: { error }` surfaced, rather than discarding five specialists' worth of paid completed work. `dedupeFindings` (exact-string) still runs first; the reconciler catches what it can't.
- **Inputs per the work order:** all findings, sheet inventory (numbers/titles/disciplines), gapCheck gaps, multi-product sheet list — plus the brand-tagged `requiredConditionsUsed` rows from the specialists, so escalation judgments are grounded on actual `where_expected` values rather than the model's guess. `runFromPdf.js` now passes `sheetInventory` (from extraction's title/discipline fields) and `gapCheck` into `runProjectSynthesis`; direct calls (hand-typed test sets) get a numbers-only inventory derived from the input sheets and a "Not available" gap check — handled, per the work order's note.
